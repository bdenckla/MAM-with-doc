Review of Chabad’s CTR

The Chabad website has an edition of the Hebrew Bible called The Complete Tanach with Rashi (CTR). (See my document, “On the Provenance of Chabad’s CTR.”)

This CTR is certainly the weirdest Hebrew Bible on the web, and possibly the worst. Whether it is the worst depends somewhat on (1) taste and (2) what purpose CTR would be put to.

For example, this CTR, though full of what most people would consider errors, might still be correct in some cases where other editions are wrong. In other words, its very weirdness, though usually a negative, might in some cases be a positive.

I will not attempt to give a comprehensive review of CTR. Instead, I will review its Psalm 32. If its Psalm 32 is representative, my review should give a good idea of CTR as a whole. At a minimum, my review is likely to give a good idea of of CTR’s poetic books, since some of CTR’s issues are specific to the poetic accents.

First, I will discuss the ways in which CTR’s Psalm 32 differs systematically from other editions. Then I will discuss the particular (i.e. non-systematic) ways in which CTR’s Psalm 32 differs from other editions.

Systematically, CTR’s Psalm 32 is restricted to the code points in Unicode 2.0. There are three code points that were introduced later that are used in some editions of Psalm 32:

  • QAMATS QATAN
  • ATNAH HAFUKH
  • HOLAM HASER FOR VAV

In the first two of the three cases, the fact that CTR does not use the newer code point causes no problems. Indeed many editions, even quite recent ones, are like CTR in that they do not make the following distinctions:

  • qamats vs. qamats qatan
  • yeraḥ ben yomo vs. atnaḥ hafukh

Both distinctions were introduced in Unicode 4.1, although ATNAH HAFUKH was introduced in a way that makes its adoption difficult, even to this day. See my proposal to the Unicode SEWG.

The third post-2.0 code point that CTR might have used is HOLAM HASER FOR VAV. In this case, the fact that CTR does not use it causes a problem. It causes a problem because CTR does not merely decline to make the distinction that this code point enables. Rather, CTR uses a nonstandard workaround to try to make this distinction. This workaround may work (accidentally or intentionally) in some fonts, but not in all fonts. The workaround is to use ZWJ followed by HOLAM instead of HOLAM HASER FOR VAV.

CTR even declines to avail itself of distinctions that were available in Unicode 2.0 For example, CTR does not distinguish tsinnor from tsinnorit, using only the ZARQA code point for both. To be fair, the annotations and names of the ZARQA and ZINOR code points were, in my opinion, botched, and that could explain why CTR uses only ZARQA. According to the annotation for ZARQA, it should be used for both tsinnor and tsinnorit if an above-center (rather than above-left) placement of tsinnor is desired. In my opinion, this annotation encourages the use of Unicode to make a typographic choice that should be left up to fonts, not “baked in” to the encoding. (See my proposal to the Unicode SEWG.)

In several cases, CTR declines to avail itself of distinctions that were available in Unicode 2.0, but, curiously, makes these distinctions in nonstandard ways. On a letter with a vowel mark (including HOLAM), CTR distinguishes prepositives from their impositive “lookalikes” by their encoded position relative to the vowel:

  • The code point TIPEHA codes both deḥi (!) and tarḥa / tipeḥa.
  • The code point GERESH codes both geresh muqdam (!) and geresh.
  • The code point YETIV codes both yetiv and mahapakh (!).

In detail:

  • The code point TIPEHA codes both deḥi (!) and tarḥa / tipeḥa.
    • The code point TIPEHA before a vowel codes deḥi (!), e.g. to encode the segol and deḥi under the letter he in הֶ֭חרשתי in Psalm 32:3 (fully: הֶ֭חֱרַשְׁתִּי ),
      • Instead of ‹POINT SEGOL, DEHI›, CTR uses ‹TIPEHA, POINT SEGOL›.
      • Schematically (using a plus sign to indicate a kind of concatenation), instead of הֶ+ה֭, CTR uses ה֖+הֶ.
      • This is very unlikely to have the desired appearance in most fonts. Plus, in normalizing contexts like most web browsers, the font won’t even get a chance to try. E.g. this document’s context, it will look like this: הֶ֖חרשתי.
      • Above we have ignored a gaʿya that CTR also has under the he. We have ignored it because (1) it is not relevant to the issue at hand and (2) it is one of many “extra” gaʿya marks that CTR has compared to many other editions.
    • The code point TIPEHA after a vowel codes tarḥa / tipeḥa, e.g. ???
  • The code point GERESH codes both geresh muqdam (!) and geresh.
    • The code point GERESH before a vowel codes geresh muqdam (!), e.g. ???
    • The code point GERESH after a vowel codes geresh, e.g. ???
  • The code point YETIV codes both yetiv and mahapakh (!).
    • The code point YETIV before a vowel codes yetiv, e.g. ???
    • The code point YETIV after a vowel codes mahapakh (!), e.g. to encode the qamats and mahapakh under the letter tav in אתָּ֤ה in Psalm 32:7 (fully: אַתָּ֤ה׀),
      • Instead of ‹QAMATS, MAHAPAKH›, CTR uses ‹QAMATS, YETIV›.
      • Schematically (using a plus sign to indicate a kind of concatenation), instead of תָּ+תּ֤, CTR uses תָּ+תּ֚.
      • This is very unlikely to have the desired appearance in most fonts. Plus, in normalizing contexts like most web browsers, the font won’t even get a chance to try. E.g. this document’s context, it will look like this: אתָּ֚ה.

One might naturally wonder how, on a letter without a vowel mark, CTR represents the six accents of these three “lookalike” pairs.

  • TIPEHA again codes both deḥi and tarḥa: XXX ambiguously?!
  • GERESH again codes both geresh muqdam and geresh: but these accents are exclusive to the poetic and prose systems respectively, so there is no ambiguity.
  • YETIV again codes both yetiv and mahapakh, but, with no pattern I can discern, sometimes MAHAPAKH codes mahapakh, as in Ps 32:11 שמח֤ו.
